# TruckMind-Agent 理论思路与方案设计报告

## 1. 核心结论

TruckMind-Agent 表面上是一个“大模型 Agent 连续决策”问题，但从优化结构看，它更接近：

> **带时间窗、空间移动成本、动态货源可见性的信息受限在线路径优化问题。**

因此，最合适的方案不是让 LLM 每一步直接决定接哪一单，而是构建一个：

> **主动感知的自适应机会成本规划器：Active-sensing Adaptive Opportunity Planner, AOP。**

它的基本思想是：

```text
每轮通过环境 API 获取有限货源信息；
主动选择少量查询点扩大感受野；
对可见货源进行精确本地仿真；
用“净收益 - 时间机会成本 + 目的地价值”打分；
选择接单、等待或少量战略空驶。
```

LLM 不作为底层优化器，而作为偏好解析、策略解释、异常兜底模块。**偏好约束是核心组成部分，不是可选项**——D006-D010 的偏好罚分可达数千元，必须在 P0 阶段就纳入硬规则过滤。LLM 在系统中的首要职责是将自然语言偏好转为结构化约束模板，供 Planner 做动作过滤和罚分估计。

## 2. 系统约束与接口事实

### 2.1 固定交互入口

评测主循环调用的是 Agent 的固定接口：

```python
decide(driver_id: str) -> dict[str, Any]
```

Agent 内部可以调用三个环境能力：

```python
get_driver_status(driver_id)
query_cargo(driver_id, latitude, longitude)
model_chat_completion(payload)
```

这些接口由 `ports.py` 中的 `AgentDecisionPort` 和 `SimulationApiPort` 约定。也就是说，提交方案应保留 `ModelDecisionService.decide(driver_id)` 这个入口，而不是重写 server 或仿真主循环。([GitHub][1])

---

### 2.2 三类动作

Agent 每轮最终只能返回三类动作之一：

```json
{"action": "take_order", "params": {"cargo_id": "..."}}
{"action": "reposition", "params": {"latitude": ..., "longitude": ...}}
{"action": "wait", "params": {"duration_minutes": ...}}
```

仓库 README 也明确说明了三类动作：接单、空驶、休息；最终评分公式为：

```text
net_income = gross_income - distance_km × cost_per_km - preference_penalty
```

因此，核心就是最大化：

```text
gross_income - distance_km × cost_per_km - preference_penalty
```

其中 `preference_penalty` 对 D006-D010 可能高达数千元，**必须从 P0 阶段就纳入约束**，而非延后处理。

([GitHub][2])

---

### 2.3 查询货源会真实消耗仿真时间

`query_cargo(driver_id, latitude, longitude)` 不是免费信息获取。它会查询指定位置附近最多 100 条当前 online 货源，然后按返回条数收取浏览耗时，默认每 10 条货源消耗 1 分钟。这个逻辑在 `EmbeddedDecisionEnvironment.query_cargo()` 中实现：先调用 `simulation_actions.query_cargo()`，再调用 `apply_cargo_query_scan_cost()` 推进时间。([GitHub][3])

因此，决策流程必须是：

```python
status0 = api.get_driver_status(driver_id)

cargo_resp = api.query_cargo(
    driver_id,
    status0["current_lat"],
    status0["current_lng"],
)

# query_cargo 会推进仿真时间，所以必须重新取状态
status1 = api.get_driver_status(driver_id)
```

这点非常关键。若使用 query 前的旧时间进行接单模拟，会错误判断装货时间窗、订单有效性和完单时间。

---

### 2.4 查询位置可以不是司机当前位置

`query_cargo` 的输入是任意 `latitude, longitude`。底层 `CargoRepository.nearest_pickup_km()` 会在当前 online 货源池中，以该查询点为中心返回最近的至多 `k` 条货源。也就是说，查询点只是“信息探针”，不会移动司机；司机位置只有 `take_order` 或 `reposition` 才会改变。([GitHub][4])

这意味着我们可以主动查询：

```text
当前位置附近货源
候选订单终点附近货源
历史高价值热区附近货源
准备空驶目标点附近货源
```

这会扩大感受野，但代价是消耗仿真时间。

---

### 2.5 接单动作内部已经包含空驶到装货点

`take_order()` 的真实流程是：

```text
当前位置 → 空驶到货源起点 → 等待装货时间窗 → 干线运输 → 到达货源终点
```

代码中会计算当前位置到货源起点的 Haversine 距离，按空驶速度转换成分钟；若早于装货窗开始，则等待；若晚于装货窗结束，则接单失败；成功后按 `cost_time_minutes` 推进到卸货点。([GitHub][5])

因此，不应该为了接一个已经可见的订单而先 `reposition` 到它的起点。直接 `take_order` 即可，因为接单本身已经包含这段空驶。

---

## 3. 问题理论定位

### 3.1 不是纯序列决策，而是动态可见图优化

如果所有货源一开始完全可见，这个问题可以抽象为：

```text
节点：订单
边：从一个订单完成状态转移到另一个订单起点
边权：收入 - 空驶成本 - 干线成本
约束：时间窗、订单有效期、月度 horizon
目标：选择一条收益最大的订单路径
```

这类似：

```text
Orienteering Problem with Time Windows
Vehicle Routing Problem with Time Windows
Resource-Constrained Longest Path
```

但 TruckMind 的实际接口并不提供全局货源。Agent 必须通过 `query_cargo` 获取当前 online 的局部货源信息，而且查询本身会推进时间。赛题文档也说明，除每轮最小必要输入外，其余状态与货源信息应通过环境规定的交互方式取得，不能绕过规定机制直接篡改环境内部状态。([GitHub][6])

所以更准确的定位是：

> **信息受限的动态在线路径优化问题。**

---

### 3.2 为什么不是直接 RL

可以把它形式化成 MDP：

```text
state = 当前时间、位置、已知货源、司机状态
action = take_order / wait / reposition / query
reward = 收入 - 成本 - 偏好罚分
```

但直接 RL 不适合作为当前第一方案：

```text
状态空间大
动作空间大
奖励延迟长
训练样本少
仿真成本高
很难保证不违反硬约束
```

更现实的路线是：

```text
显式仿真规则 + 启发式在线优化 + 少量大模型辅助
```

---

## 4. 方案总览：Active-sensing Adaptive Opportunity Planner

方案名称：

> **AOP：主动感知的自适应机会成本规划器**

整体框架：

```text
ModelDecisionService.decide(driver_id)
    ↓
读取当前司机状态
    ↓
主动选择查询点 query_cargo
    ↓
更新货源缓存与区域热度
    ↓
对候选货源做精确本地仿真
    ↓
计算 utility = 净收益 - 时间机会成本 + 目的地价值
    ↓
输出 take_order / wait / reposition
```

核心思想：

```text
不追求一次性全局最优；
每轮在有限可见信息下做最优近似；
通过主动查询扩大感受野；
通过机会成本避免接低效率订单；
通过缓存和热区估计弥补局部查询的短视。
```

---

## 5. 候选订单精确仿真

每个候选货源都要本地模拟一遍，得到真实可执行收益。

### 5.1 输入

```text
当前时间 current_time
当前位置 current_lat/current_lng
货源 start/end
货源 load_time
货源 cost_time_minutes
货源 price
司机 cost_per_km
空驶速度 speed_kmph
```

### 5.2 计算过程

```text
pickup_km = 当前点到货源起点距离

pickup_minutes = ceil(pickup_km / speed_kmph * 60)

arrival_time = current_time + pickup_minutes

如果 arrival_time > load_window_end:
    货源不可行

ready_time = max(arrival_time, load_window_start)

wait_minutes = ready_time - arrival_time

finish_time = ready_time + cost_time_minutes

haul_km = 货源起点到终点距离

net_profit = price - cost_per_km * (pickup_km + haul_km)

duration_hours = (pickup_minutes + wait_minutes + cost_time_minutes) / 60
```

### 5.3 过滤条件

候选订单直接过滤：

```text
1. 赶不上装货时间窗
2. 完单超过仿真 horizon
3. 已经过期或已接过
4. 净收益明显为负
5. duration 过长且单位收益过低
```

---

## 6. 核心评分函数：机会成本 Utility

单看净收益不够，因为时间是有限资源。一个订单虽然净收益为正，但如果占用太久，可能错过更好的机会。

因此使用：

```text
utility(order)
=
net_profit(order)
- λ × duration_hours(order)
+ β × destination_value(order.end)
```

其中：

```text
net_profit = price - cost_per_km × (pickup_km + haul_km)
λ = 当前每小时机会成本
destination_value = 订单终点附近未来接单价值
```

### 6.1 λ 的含义

`λ` 表示：

> 当前阶段，一小时仿真时间大约值多少钱。

如果一单占用 5 小时，而当前机会成本是 300 元/小时，那么这单至少要贡献约 1500 元净利润才值得接。

### 6.2 λ 的估计方式

#### 6.2.1 问题：纯分位数估计不收敛

直接用 `percentile(observed_profit_per_hour, 60)` 存在问题：

```text
仿真初期，observed 样本极少，分位数不稳定
前几单低收益 → λ 被低估 → 后续"饥不择食"
前几单高收益 → λ 被高估 → 后续"过于挑剔"
```

#### 6.2.2 改进：贝叶斯先验 + 经验更新

用先验-经验混合估计，样本越多越信任经验：

```python
# 先验：基于离线数据分析的基准值（见第17节离线分析）
prior_lambda = 150  # 元/小时，可从 cargo_dataset.jsonl 统计得到

# 随观测更新
if len(observed) < 10:
    lambda_hour = prior_lambda
else:
    empirical = percentile(observed, 60)
    # 混合先验和经验，样本越多越信任经验
    alpha = min(len(observed) / 50, 0.8)
    lambda_hour = (1 - alpha) * prior_lambda + alpha * empirical

lambda_hour = clamp(lambda_hour, min_value=50, max_value=500)
```

#### 6.2.3 时段区分

不同时段的 λ 应该不同，因为货源密度和收益分布不同：

```python
# 白天（08:00-20:00）：货源多，机会成本高
# 夜间（20:00-08:00）：货源少，机会成本低（但偏好可能禁止接单）
hour = get_current_hour(status)
if 8 <= hour < 20:
    time_multiplier = 1.0
else:
    time_multiplier = 0.7  # 夜间机会成本降低

lambda_hour *= time_multiplier
```

#### 6.2.4 剩余时间调整

随剩余时间调整，月末适当降低阈值：

```python
remaining_ratio = remaining_minutes / horizon_minutes
lambda_hour = base_lambda * (0.7 + 0.6 * remaining_ratio)
```

直觉：

```text
月初时间充足，挑单更严格；
月末时间即将耗尽，阈值适当降低。
```

## 7. 主动查询策略：扩大感受野

### 7.1 查询点选择的本质

因为 `query_cargo` 可以查询任意位置附近的货源，所以查询点选择本质是：

> **信息获取预算分配问题。**

每次 query 的价值是：

```text
query_value(point)
=
该查询可能改善当前决策的收益
- 查询消耗时间对应的机会成本
```

查询成本：

```text
query_cost_minutes = ceil(returned_items_count / 10)
query_cost_value = λ × query_cost_minutes / 60
```

---

### 7.2 查询点优先级

推荐优先级如下：

| 优先级 | 查询点            | 作用             |
| --- | -------------- | -------------- |
| P0  | 当前司机位置         | 基础候选集，必须查      |
| P1  | 当前 top1 订单终点   | 估计接完这单后周边是否有好货 |
| P2  | top2/top3 订单终点 | 当候选分数接近时用于比较   |
| P3  | 历史高价值热区        | 当前无好货时寻找机会     |
| P4  | 当前候选货源起点聚类中心   | 扩大附近局部感受野      |
| P5  | 探索型网格点         | 暂不建议第一版使用      |

---

### 7.3 每轮查询预算

第一版建议：

```text
每轮最多 3 次 query：
1. 当前位置，必查
2. top1 订单终点，条件触发
3. top2 终点或最佳热区，条件触发
```

不要无限多查，因为每次查询都会推进仿真时间。`SimulationOrchestrator` 会把决策阶段 query 消耗计入 `query_scan_cost_minutes`，再加上动作执行耗时，形成整步真实推进时间。([GitHub][7])

---

### 7.4 触发额外查询的条件

#### 情况一：候选订单分数接近

如果当前 top1 和 top2 分数差很小：

```text
best_score - second_score < threshold
```

说明目的地价值可能改变选择，值得查询 top1/top2 终点。

#### 情况二：top1 当前收益高，但终点未知

如果 top1 净收益高，但终点区域从未查过：

```text
query(top1.end)
```

避免接到“当前赚钱但终点荒芜”的订单。

#### 情况三：当前区域无好货

如果当前位置 query 后没有 `utility > 0` 的订单，可以查询历史最佳热区：

```text
query(best_hotspot)
```

若热区价值足够高，再考虑 `reposition`；否则 `wait`。

#### 情况四：准备 reposition 前

如果要空驶到某个目标点，必须先查目标点附近货源：

```text
不查询目标点，不进行 reposition
```

因为 reposition 有真实成本和时间成本，不能盲目移动。

## 8. 区域价值与缓存系统

### 8.1 货源缓存

每次通过 `query_cargo` 看到的货源都缓存下来：

```python
cargo_cache[cargo_id] = {
    "cargo": cargo,
    "first_seen_time": current_time,
    "last_seen_time": current_time,
    "query_point": point,
}
```

候选集不只来自本轮 query，也可以来自缓存中仍然有效的货源：

```text
candidate_set = current_query_items + valid_cached_items
```

因为这些信息是通过合法 API 获得的，不是绕过环境读取全量数据。

### 8.1.1 缓存有效性精确定义

缓存货源必须同时满足以下条件才算有效：

```python
def is_cache_valid(cached, current_time_minutes, taken_cargo_ids):
    """缓存货源是否仍然有效"""
    cargo = cached["cargo"]

    # 1. 未过期：remove_time 还没到
    if cargo["remove_minutes"] < current_time_minutes:
        return False

    # 2. 未被接走：cargo_id 不在已接列表中
    if cargo["cargo_id"] in taken_cargo_ids:
        return False

    # 3. 装货窗还没过：当前时间还能赶上装货
    #    需要估算从当前位置到装货点的空驶时间
    #    简化：只要 load_time_end 还没过就算有效
    if current_time_minutes > cargo["load_time_end_minutes"]:
        return False

    # 4. 缓存不过旧：距离上次看到不超过一定时间
    #    防止使用过时信息（货源可能已被其他机制移除）
    cache_age = current_time_minutes - cached["last_seen_time"]
    if cache_age > 360:  # 6小时
        return False

    return True
```

注意：当前仿真中每个司机独立运行，`taken_cargo_ids` 只需记录自己接过的单。但 `remove_time` 和 `load_time` 必须检查，因为 `CargoRepository.sync_time_minutes()` 会让过期货源下线。

---

### 8.2 区域热度 Heatmap

把地图离散成 grid：

```python
grid = (round(lat, 2), round(lng, 2))
```

对每个区域维护：

```text
观察货源数量
平均 profit_per_hour
top-k profit_per_hour
最近一次查询时间
```

区域价值可定义为：

```text
heatmap_value(grid)
=
top-k profit_per_hour 的平均值
```

---

### 8.3 目的地价值 Destination Value

对于某个订单终点 `end`，如果查询过该点附近货源，可以估计：

```text
destination_value(end)
=
从 end 位置出发，在 finish_time 附近可接货源的 top-k utility 平均值
```

更具体：

```python
def estimate_region_value(location, items, future_time):
    values = []
    for item in items:
        ev = simulate_from_location(location, future_time, item)
        if ev.feasible:
            values.append(ev.utility)
    return mean(top_k(values, k=5))
```

这相当于对价值函数做局部估计：

```text
V(location, time)
```

于是订单最终评分变成：

```text
score(order)
=
current_net_profit
- λ × duration_hours
+ β × V(order.end, finish_time)
```

## 9. wait 与 reposition 策略

### 9.1 wait 策略

#### 9.1.1 核心洞察：短 wait + 重新 query 优于长 wait

wait 的机会成本很高（什么都不做），而 query 的成本相对较低（1-10 分钟）。长 wait（如 180 分钟）会错过期间上线的新货源。更好的策略是：

```text
短 wait → 重新 query → 发现新货源 → 接单
```

因为 `CargoRepository.sync_time_minutes()` 会让新货源上线，wait 后再 query 能发现新机会。

#### 9.1.2 wait 时长分档

```python
def smart_wait_duration(status, memory, constraints):
    """根据当前状态和偏好约束决定 wait 时长"""
    remaining = horizon_minutes - status["simulation_progress_minutes"]
    current_hour = get_current_hour(status)

    # 1. 偏好驱动的 wait（最高优先级）
    pref_wait = preference_aware_wait(driver_id, status, memory, constraints)
    if pref_wait is not None:
        return pref_wait

    # 2. 仿真末期：wait 到结束
    if remaining < 60:
        return remaining

    # 3. 常规策略：短 wait + 重新 query
    if items_count == 0:
        # 当前无货，等新货源上线，30-60 分钟后重新 query
        wait_minutes = 30
    elif best_utility < -100:
        # 有货但质量差，等更好的货源
        wait_minutes = 30
    elif best_utility < 0:
        # 接近阈值，短等一下
        wait_minutes = 15
    else:
        # 有好货但被偏好过滤了，等偏好窗口开放
        wait_minutes = 15

    # 4. 不要 wait 超过剩余时间
    wait_minutes = min(wait_minutes, remaining)
    return wait_minutes
```

#### 9.1.3 wait 后的 query 策略

wait 之后，下一轮 decide() 会自动重新 query 当前位置，相当于：

```text
wait 30 分钟 → query 当前位置（1-10 分钟）→ 评估新货源
```

总消耗约 31-40 分钟，但能发现新上线的货源。比 wait 180 分钟更灵活。

---

### 9.2 reposition 策略

暂不考虑偏好时，`reposition` 应非常克制。

原因：

```text
take_order 已经包含到货源起点的空驶；
reposition 本身没有收入；
如果没有可靠热区信息，空驶很容易亏损。
```

只有满足以下条件才 reposition：

```text
1. 当前区域连续多轮无好货
2. 已查询目标热区，确认有较高区域价值
3. 移动成本和时间机会成本小于预期收益提升
4. 目标点不是随便生成的经纬度，而是高价值热区中心
```

判断公式：

```text
move_value
=
heatmap_value(target) × expected_active_hours
- cost_per_km × distance_to_target
- λ × travel_hours
```

只有当：

```text
move_value > wait_value + margin
```

才空驶。

## 9.3 错误恢复机制

### 9.3.1 take_order 失败处理

从代码看，`_apply_action` 中 take_order 失败（cargo 已过期/被接走）会消耗 1 分钟并返回 `accepted=False`，但 Agent 不会收到这个反馈。

因此必须在返回 action 前做本地校验：

```python
def safe_take_order(candidates, status, memory):
    """选择第一个仍然有效的候选订单"""
    for candidate in sorted_candidates:  # 按 utility 降序
        if is_cache_valid(candidate, status["simulation_progress_minutes"],
                          memory.taken_cargo_ids):
            return {"action": "take_order", "params": {"cargo_id": candidate.cargo_id}}
    # 全部无效，fallback to wait
    return {"action": "wait", "params": {"duration_minutes": 15}}
```

### 9.3.2 异常兜底

如果 decide() 内部抛异常，server 会怎样？从代码看，`_call_agent` 只做格式检查，异常会导致该 step 失败。因此：

```python
def decide(self, driver_id: str) -> dict[str, Any]:
    try:
        return self._decide_inner(driver_id)
    except Exception as e:
        # 兜底：安全 wait，不崩溃
        return {"action": "wait", "params": {"duration_minutes": 30}}
```

## 10. 是否需要 Beam Search

Beam Search 可以作为增强，但不应是第一优先级。

原因是当前 query 返回的是“查询点附近当前 online 的最近 100 条货源”。如果接完第一单，司机到了第一单终点，真正 relevant 的第二步货源应该是终点附近的货源，而这些未必在当前 query 结果里。`CargoRepository.nearest_pickup_km()` 的确是以查询点为中心取最近货源，因此只在当前位置 query 上做深度搜索会有视野偏差。([GitHub][4])

建议实现小 Beam：

```text
depth = 2
beam_width = 5
candidate_k = 20
```

只用于当前缓存候选集中的局部路线修正：

```text
如果当前候选里存在可以连续衔接的订单链，就避免选一个破坏后续链条的订单。
```

但第一版核心仍然是：

```text
精确单票仿真 + 主动查询 + 机会成本阈值
```

## 11. LLM 的合理作用：偏好解析与结构化约束

LLM 不适合直接做底层动作决策。它不稳定计算距离、时间窗、机会成本，也无法可靠维护月度状态。

但 LLM 在本系统中有一个**不可替代的核心作用**：将自然语言偏好文本转为结构化约束模板。

### 11.1 偏好解析的必要性

D006-D010 的偏好是自然语言描述，例如：

```text
D006: "每天至少连续休息5小时"
D007: "23:00-04:00不接单、不空驶"
D008: "每月至少2天完全不出车"
D009: "每天23:00前回家，次日08:00前不再接单/空驶"
D010: "每月至少5天到目标点；禁止进入指定区域"
```

这些偏好如果违反，罚分可达数千元。**必须从 P0 阶段就纳入硬规则过滤**。

### 11.2 LLM 偏好解析流程

每个司机初始化时调用一次 LLM，将偏好文本转为结构化约束：

```text
preferences text (from drivers.json)
    ↓
LLM parser (一次性调用)
    ↓
structured constraint template (JSON)
    ↓
Planner 使用约束做动作过滤和罚分估计
```

### 11.3 结构化约束模板设计

LLM 输出的约束模板应包含以下类型：

```python
@dataclass
class PreferenceConstraint:
    driver_id: str
    rules: list[Rule]

@dataclass
class Rule:
    rule_type: str  # "night_rest" | "daily_rest" | "monthly_off" | "home_before" | "must_visit" | "forbidden_area"
    params: dict    # 规则参数
    penalty_type: str  # "per_violation" | "lump_sum"
```

具体模板示例：

```python
# D006: 每天至少连续休息5小时
{
    "rule_type": "daily_rest",
    "params": {
        "min_continuous_rest_hours": 5,
        "rest_action": "wait"  # 只有 wait 算休息
    }
}

# D007: 23:00-04:00 不接单、不空驶
{
    "rule_type": "night_rest",
    "params": {
        "forbidden_hours": [23, 0, 1, 2, 3],  # 23:00-04:00
        "forbidden_actions": ["take_order", "reposition"]
    }

# D008: 每月至少2天完全不出车
{
    "rule_type": "monthly_off",
    "params": {

    }

    "params": {
```

### 11.5 硬规则过滤：Planner 如何使用约束

解析后的约束在 Planner 中以**硬规则过滤器**的形式使用：

```python
def passes_preference_rules(driver_id, action, status, memory, constraints):
    """硬规则：违反偏好的动作直接拒绝，不进入评分"""
    for rule in constraints.rules:
        if rule.rule_type == "night_rest":
            # 检查动作执行时段是否在禁止时间内
            if action["action"] in rule.params["forbidden_actions"]:
                exec_hour = get_execution_hour(status, action)
                if exec_hour in rule.params["forbidden_hours"]:
                    return False

        elif rule.rule_type == "daily_rest":
            # 检查今天是否已有足够连续 wait
            if not has_sufficient_daily_rest(memory, rule.params["min_continuous_rest_hours"]):
                # 如果今天还没休息够，且当前时段适合休息，优先 wait
                pass  # 不硬拒绝，但在评分中大幅降权

        elif rule.rule_type == "monthly_off":
            # 检查本月已有几天完全不出车
            off_days = count_off_days(memory)
            remaining_days = get_remaining_days(status)
            if off_days < rule.params["min_off_days"]:
                # 如果休息天数不够，某些天必须强制休息
                if should_force_off_day(status, memory, off_days, remaining_days):
                    return action["action"] == "wait"  # 只允许 wait

        elif rule.rule_type == "home_before":
            # 检查是否到了该回家的时间
            current_hour = get_current_hour(status)
            if current_hour >= rule.params["home_hour"] - 1:
                if action["action"] in rule.params["forbidden_actions_after_home"]:
                    return False

        elif rule.rule_type == "forbidden_area":
            # 检查动作目标是否在禁止区域内
            target = get_action_target(action)
            if point_in_polygon(target, rule.params["polygon"]):
                return False

    return True
```

### 11.6 偏好感知的 wait 策略

偏好约束不仅过滤动作，还影响 wait 决策：

```python
def preference_aware_wait(driver_id, status, memory, constraints):
    """根据偏好约束决定是否需要主动 wait 来满足偏好"""
    for rule in constraints.rules:
        if rule.rule_type == "night_rest":
            current_hour = get_current_hour(status)
            if current_hour >= rule.params["forbidden_hours"][0] - 1:
                # 接近夜间禁运时段，wait 到次日早上
                wait_until = next_morning_hour(status, rule.params)
                return {"action": "wait", "params": {"duration_minutes": wait_until}}

        elif rule.rule_type == "daily_rest":
            if not has_sufficient_daily_rest(memory, rule.params["min_continuous_rest_hours"]):
                # 今天还没休息够，强制 wait 补足
                needed = rule.params["min_continuous_rest_hours"] * 60 - get_today_rest_minutes(memory)
                if needed > 0:
                    return {"action": "wait", "params": {"duration_minutes": needed}}

        elif rule.rule_type == "monthly_off":
            off_days = count_off_days(memory)
            remaining_days = get_remaining_days(status)
            min_off = rule.params["min_off_days"]
            if off_days < min_off and remaining_days <= min_off - off_days + 1:
                # 剩余天数紧张，今天必须休息
                return {"action": "wait", "params": {"duration_minutes": 1440}}

    return None  # 无需偏好驱动的 wait
```

### 11.7 Token 成本分析

偏好解析的 token 消耗极低：

```text
每个司机初始化时调用一次 LLM
一次调用约 500-2000 tokens
10 个司机 = 5,000-20,000 tokens
远低于复赛 5M tokens/driver 限制
```

因此，**偏好解析应在系统启动时完成，不在每轮 decide() 中调用 LLM**。

### 11.8 LLM 的其他辅助作用

除偏好解析外，LLM 还可用于：

```text
1. 策略模式解释（赛后报告）
2. 异常场景兜底说明
3. 可解释性输出
```

但**核心决策逻辑不依赖 LLM**，确保系统稳定性和确定性。

### 12.3 主流程伪代码

```python
class ModelDecisionService:
    def __init__(self, api: SimulationApiPort):
        self._api = api
        self.memory_by_driver: dict[str, DriverMemory] = {}
        self.constraints_by_driver: dict[str, PreferenceConstraint] = {}
        self._initialized = False

    def _ensure_initialized(self, driver_id: str):
        """首次遇到某司机时，解析偏好并初始化状态"""
        if driver_id not in self.memory_by_driver:
            # 加载离线先验数据
            prior = load_prior_data("prior_data.json")
            self.memory_by_driver[driver_id] = DriverMemory(driver_id, prior)

            # LLM 偏好解析（一次性调用）
            status = self._api.get_driver_status(driver_id)
            prefs_text = status.get("preferences", "")
            if prefs_text:
                self.constraints_by_driver[driver_id] = (
                    self.preference_parser.parse(prefs_text)
                )
            else:
                self.constraints_by_driver[driver_id] = PreferenceConstraint(
                    driver_id=driver_id, rules=[]
                )

    def decide(self, driver_id: str) -> dict:
        try:
            return self._decide_inner(driver_id)
        except Exception:
            # 异常兜底：安全 wait，不崩溃
            return {"action": "wait", "params": {"duration_minutes": 30}}

    def _decide_inner(self, driver_id: str) -> dict:
        self._ensure_initialized(driver_id)
        memory = self.memory_by_driver[driver_id]
        constraints = self.constraints_by_driver[driver_id]

        # 1. 偏好驱动的 wait（最高优先级）
        pref_wait = preference_aware_wait(driver_id, status_hint, memory, constraints)
        if pref_wait is not None:
            memory.record_action(pref_wait)
            return pref_wait

        # 2. 获取 query 前状态
        status0 = self._api.get_driver_status(driver_id)

        # 3. 当前位置必查
        items0 = self._api.query_cargo(
            driver_id,
            float(status0["current_lat"]),
            float(status0["current_lng"]),
        )

        # 4. query 后重新读取状态（query 推进了时间！）
        status = self._api.get_driver_status(driver_id)

        # 5. 更新缓存和 heatmap
        memory.update_query_result(items0, status)

        # 6. 初步评估候选订单（含偏好过滤）
        evals = self.evaluate_candidates(status, memory, constraints)

        # 7. 选择战略查询点（含 ROI 判断）
        query_points = self.query_policy.select_points(
            status=status,
            evals=evals,
            memory=memory,
            max_extra_queries=2,
        )

        # 8. 执行额外查询
        for p in query_points:
            extra_items = self._api.query_cargo(driver_id, p.lat, p.lng)
            status = self._api.get_driver_status(driver_id)
            memory.update_query_result(extra_items, status)

        # 9. 时间已推进，重新评估
        final_evals = self.evaluate_candidates(status, memory, constraints)

        # 10. 仿真末期过滤
        remaining = horizon_minutes - status["simulation_progress_minutes"]
        if remaining < 720:
            final_evals = [e for e in final_evals
                           if e.total_minutes < remaining - 60]

        # 11. 决策：选最优且有效的候选
        for candidate in sorted(final_evals, key=lambda x: -x.utility):
            if candidate.utility > 0 and is_cache_valid(
                candidate, status["simulation_progress_minutes"],
                memory.taken_cargo_ids
            ):
                action = {
                    "action": "take_order",
                    "params": {"cargo_id": candidate.cargo_id},
                }
                memory.record_action(status, action)
                return action

        # 12. 无好货，短 wait + 重新 query
        wait_min = smart_wait_duration(status, memory, constraints)
        action = {"action": "wait", "params": {"duration_minutes": wait_min}}
        memory.record_action(status, action)
        return action
```

## 13. 离线数据分析：获取先验知识

虽然正式方案以 API-only 为主，但离线分析 `cargo_dataset.jsonl` 和 `drivers.json` 可以为在线规划器提供**先验知识**，显著提升初期决策质量。

### 13.1 可提取的先验信息

```text
1. 货源时空分布统计
   - 每小时新上线货源数量
   - 各区域货源密度
   - 货源价格分布（均值、分位数）

2. 区域价值先验
   - 各 grid 的平均 profit_per_hour
   - 高价值热区坐标
   - 低价值区域（应避免空驶前往）

3. λ 先验值
   - 全局 profit_per_hour 的 60 分位数
   - 分时段 profit_per_hour
   - 不同区域的 profit_per_hour

4. 货源上线时间模式
   - 一天中哪些时段新货源最多
   - 货源平均有效期
   - 高价值货源的上线规律

5. 司机成本参数
   - 各司机的 cost_per_km
   - 偏好规则的具体参数
```

### 13.2 离线分析脚本

```python
# offline_analysis.py
# 在评测环境外运行，生成先验数据文件

import json
import numpy as np
from collections import defaultdict

def analyze_cargo_dataset(path):
    """分析货源数据集，提取先验统计"""
    cargos = []
    with open(path) as f:
        for line in f:
            cargos.append(json.loads(line))

    # 1. 价格分布
    prices = [c["price"] / 100 for c in cargos]  # 分→元
    print(f"价格: mean={np.mean(prices):.0f}, p50={np.percentile(prices, 50):.0f}, p90={np.percentile(prices, 90):.0f}")

    # 2. 区域密度
    grid_counts = defaultdict(int)
    grid_profits = defaultdict(list)
    for c in cargos:
        start = c["start"]
        grid = (round(start["lat"], 1), round(start["lng"], 1))
        grid_counts[grid] += 1
        # 估算 profit_per_hour
        if c.get("cost_time_minutes"):
            pph = (c["price"] / 100) / (c["cost_time_minutes"] / 60)
            grid_profits[grid].append(pph)

    # 3. λ 先验
    all_pph = [pph for profits in grid_profits.values() for pph in profits]
    prior_lambda = np.percentile(all_pph, 60)
    print(f"λ 先验: {prior_lambda:.0f} 元/小时")

    # 4. 保存先验数据
    prior = {
        "lambda_prior": float(prior_lambda),
        "price_stats": {
            "mean": float(np.mean(prices)),
            "p50": float(np.percentile(prices, 50)),
            "p90": float(np.percentile(prices, 90)),
        },
        "top_grids": sorted(grid_counts.items(), key=lambda x: -x[1])[:50],
        "grid_avg_pph": {str(k): float(np.mean(v)) for k, v in grid_profits.items()},
    }

    with open("prior_data.json", "w") as f:
        json.dump(prior, f, ensure_ascii=False, indent=2)

    return prior
```

## 14. 实现优先级（修订版）

### P0：精确单步 Planner + 偏好硬规则 + LLM 偏好解析

目标：

```text
LLM 仅用于初始化时解析偏好文本 → 结构化约束
每轮当前位置 query
query 后重新 get status
用所有返回货源
精确模拟接单全过程
按 net_profit - λ*time 选择
偏好硬规则过滤（D006-D010）
无好货则短 wait + 重新 query
错误恢复和异常兜底
仿真末期策略
```

这是最重要的 baseline。**偏好约束从 P5 提前到 P0**，因为罚分太重不能延后。

预期收益：相对 baseline 巨大提升（baseline 完全不考虑偏好，且只用 20 条货源）。

### P1：主动查询 top1 终点 + 查询 ROI 判断

目标：

```text
每轮最多额外 query top1 订单终点
估计 destination_value
避免接到终点很差的订单
查询前做 ROI 判断：只有可能改变决策时才查
```

查询 ROI 判断：

```python
def should_query_destination(top1_eval, top2_eval, lambda_hour):
    """查询终点是否值得"""
    score_gap = top1_eval.utility - top2_eval.utility
    query_cost_value = lambda_hour * 10 / 60  # 最坏情况 10 分钟
    return score_gap < query_cost_value * 2
```

## 15. 风险与边界

### 15.1 不能直接读取全量 cargo 数据作为主方案

从代码上看，`server/data/cargo_dataset.jsonl` 可能存在于评测环境中；但赛题说明要求货源信息通过环境规定交互方式取得，并且不得绕过规定机制直接篡改或访问内部状态。([GitHub][6])

所以正式方案应以 API-only 为主，避免依赖读取全量货源文件。

---

### 15.2 多点查询会消耗时间

主动查询是优势，但不是免费优势。每次查询最多可能消耗 10 分钟。若每轮过度查询，会显著降低可运营时间。([GitHub][3])

因此必须有查询预算：

```text
当前位置必查
额外查询最多 1-2 个点
只有在可能改变决策时才查
```

---

### 15.3 Reposition 不能滥用

因为 `take_order` 已经包含去装货点的空驶，盲目 reposition 往往只是增加成本。只有当目标热区经过查询确认，并且预期收益超过移动成本和时间机会成本时，才应空驶。

---

## 16. 最终方案一句话

> TruckMind-Agent 的最佳合规技术路线，不是让 LLM 直接接单，也不是读取全量数据做离线全局规划，而是在环境 API 约束下构建一个偏好感知的主动在线调度优化器：通过 LLM 将自然语言偏好转为结构化硬约束，通过有限查询扩大感受野，通过精确仿真计算真实收益，通过机会成本控制接单阈值，通过缓存与热区估计未来价值，最终稳定输出合法、偏好合规且高收益的 `take_order / wait / reposition` 动作。

## 17. 推荐方案名称

中文：

```text
主动感知的自适应机会成本货运调度 Agent
```

英文：

```text
Active-sensing Adaptive Opportunity Planner for Truck Dispatching
```

简称：

```text
AOP Planner
```

核心卖点：

```text
1. Optimization-first，而不是 prompt-first
2. Preference-aware，偏好约束从 P0 开始，不是可选项
3. LLM-as-parser，LLM 将偏好文本转为结构化约束模板，不做底层数值优化
4. API-compatible，严格遵守仿真交互方式
5. 主动查询扩大感受野
6. 精确复刻仿真规则，保证动作可校验
7. 用机会成本解决"正收益但低效率"的订单选择问题
8. 离线先验 + 在线更新，解决冷启动问题
```
