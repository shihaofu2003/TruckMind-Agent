下面这份讲解按**评测真实运行顺序**来讲。重点不是“有哪些类”，而是：

1. **系统何时调用 Agent。**
2. **Agent 在 `decide()` 里能调用哪些 API。**
3. **每个 API 的输入、输出、内部处理，以及是否会推进仿真时间。**
4. **动作返回后，系统如何执行动作、记录日志、最后计算收益。**

---

# 0. 先给核心结论

这个 demo 不是一个外部 HTTP Agent 服务。评测时，Agent 是被**进程内 import** 进来的。

实际结构是：

```text
server/main.py
  → EvaluationRunner.run()
    → 构造 CargoRepository / DriverStateManager / ModelGatewayClient
    → build_embedded_agent_decision_engine()
      → EmbeddedDecisionEnvironment
      → ModelDecisionService(environment)
      → EmbeddedAgentDecisionEngine
    → SimulationOrchestrator.run()
      → 循环调用 embedded_engine.decide(driver_id)
```

所以你提交的核心不是 HTTP API，而是：

```python
class ModelDecisionService:
    def decide(self, driver_id: str) -> dict[str, Any]:
        ...
```

`ModelDecisionService` 通过注入的 `SimulationApiPort` 调用仿真系统能力：

```python
get_driver_status(driver_id)
query_cargo(driver_id, latitude, longitude)
model_chat_completion(payload)
```

这一层就是所谓的 **Agent API**。`ports.py` 里也明确规定了两个协议：主循环调用 `AgentDecisionPort.decide(driver_id)`，Agent 内部调用 `SimulationApiPort` 的三个方法。([GitHub][1])

# 1. 评测启动：`server/main.py`

## 1.1 输入

命令行入口大致是：

```bash
python main.py [config_path] [--max-steps N]
```

输入参数：

| 参数            | 含义                                       |
| ------------- | ---------------------------------------- |
| `config`      | 可选配置路径；不传则使用 `server/config/config.json` |
| `--max-steps` | 可选，全局最大 step 数；调试用                       |

## 1.2 输出

返回进程退出码：

| 返回值 | 含义          |
| --- | ----------- |
| `0` | 仿真正常结束      |
| `1` | 仿真过程异常，记录日志 |

## 1.3 内部处理

`main()` 做的事情很少：

```text
解析命令行参数
构造 EvaluationRunner
调用 runner.run()
异常则 logging.exception
```

也就是说，`main.py` 本身不是仿真核心，只是评测入口。代码注释也说明它是“离线仿真评测”，结果写入 `demo/results/`，无 HTTP 服务。([GitHub][2])

# 2. 配置与组件构造：`EvaluationRunner.run()`

`EvaluationRunner.run()` 是评测编排入口。

## 2.1 输入

来自构造函数：

```python
EvaluationRunner(config_path: Path | None, max_steps: int | None)
```

其中：

| 输入            | 作用           |
| ------------- | ------------ |
| `config_path` | 配置文件路径       |
| `max_steps`   | 覆盖配置里的最大仿真步数 |

## 2.2 输出

返回一个 summary 字典，包含：

```python
{
  "completed_steps": ...,
  "remaining_cargo_count": ...,
  "simulation_progress_minutes": ...,
  "simulation_wall_time": ...,
  "simulate_time_seconds": ...,
  "driver_completed_steps": ...,
  "driver_result_files": ...
}
```

## 2.3 内部处理

它按顺序做这些事：

```text
1. load_settings()
2. 配置 logging
3. CargoRepository(settings.cargo_dataset_path).load()
4. DriverStateManager(settings.drivers_path).load()
5. 构造 ModelGatewayClient
6. build_embedded_agent_decision_engine(repo, manager, model_gateway)
7. 构造 SimulationOrchestrator
8. manager.start_simulation_minutes(driver_id=None, progress_minutes=0)
9. orchestrator.run()
10. finally: model_gateway.close()
```

关键点是：这里会把同一个 `CargoRepository`、`DriverStateManager` 和 `ModelGatewayClient` 注入给 Agent 的运行环境。`EvaluationRunner` 本身不直接调用你的 Agent，而是先构造 `EmbeddedAgentDecisionEngine`，再交给 `SimulationOrchestrator`。([GitHub][3])

# 3. 配置加载：`load_settings()`

`load_settings()` 决定仿真用哪些数据、模型接口、时间参数。

## 3.1 输入

```python
load_settings(config_path: Path | None = None) -> AppSettings
```

如果不传 `config_path`，默认使用：

```text
demo/server/config/config.json
```

## 3.2 输出

返回不可变 dataclass：

```python
AppSettings(
    cargo_dataset_path,
    drivers_path,
    reposition_speed_km_per_hour,
    results_dir,
    log_dir,
    simulation_max_steps,
    simulation_duration_days,
    model_api_url,
    model_api_key,
    model_name,
    model_timeout_seconds,

```

## 3.3 内部处理

它会：

```text
1. 读取 config.json
2. 校验 cargo_dataset_path / drivers_path / speed / results_dir / max_steps 等字段
3. 优先从环境变量 DASHSCOPE_API_KEY 或 TIANCHI_MODEL_API_KEY 取模型 key
```

几个重要配置：

| `simulation_max_steps`         | 全局最大 step              |
| `simulation_duration_days`     | 仿真天数                   |
| `model_api_url`                | OpenAI-compatible 模型接口 |
| `model_name`                   | 默认模型名                  |

所以这里决定了仿真持续多久、每个动作的空驶速度、模型调用地址等。([GitHub][4])

# 4. Agent 是如何被接入的？

核心函数是：

```python
build_embedded_agent_decision_engine(repo, manager, model_gateway)
```

## 4.1 输入

```python
repo: CargoRepository
manager: DriverStateManager
model_gateway: ModelGatewayClient
```

## 4.2 输出

```python
EmbeddedAgentDecisionEngine
```

这个对象实现了：

```python
decide(driver_id: str) -> dict[str, Any]
```

因此可以被 `SimulationOrchestrator` 当成 `AgentDecisionPort` 使用。

## 4.3 内部处理

它做三件事：

```python
environment = EmbeddedDecisionEnvironment(repo, manager, model_gateway)
environment_port: SimulationApiPort = environment
decision_service = ModelDecisionService(environment_port)
return EmbeddedAgentDecisionEngine(decision_service, environment)
```

这里非常关键：

```text
你的 ModelDecisionService 不是自己 new 一个仿真环境。
它拿到的是 server 注入进来的 environment。
```

这个 environment 就是你在 `decide()` 里能调用的“Agent API”。([GitHub][5])

# 5. Agent API 一：`get_driver_status(driver_id)`

这是 Agent 获取当前司机状态的接口。

## 5.1 输入

```python
driver_id: str
```

例如：

```python
status = self._api.get_driver_status("D001")
```

## 5.2 输出

返回字典，大致结构：

```python
{
  "driver_id": "...",
  "name": "...",
  "vehicle_no": "...",
  "truck_length": "...",
  "preferences": [...],
  "current_lat": ...,
  "current_lng": ...,
  "is_current": True,
  "simulation_started": True,
  "simulation_progress_minutes": ...,
  "simulation_wall_time": "...",
  "current_order_cargo_id": ...,
  "completed_order_count": ...
}
```

## 5.3 内部处理

调用链：

```text
EmbeddedDecisionEnvironment.get_driver_status()
  → DriverStateManager.get_driver_status()
```

`DriverStateManager.get_driver_status()` 内部会：

```text
1. 根据 driver_id 找司机 profile
2. 读取当前全局仿真分钟 simulation_progress_minutes
3. 把分钟数转换成墙钟时间 simulation_wall_time
4. 返回司机当前位置、偏好、车辆信息、已完成订单数等
```

它**不会推进仿真时间**。只是读取状态。([GitHub][6])

## 5.4 需要注意

`status` 里的 `current_lat/current_lng` 是当前司机位置；`preferences` 是自然语言偏好；`simulation_progress_minutes` 是从 `2026-03-01 00:00:00` 开始经过的分钟数。

但是它没有显式返回 `cost_per_km`。成本结算脚本是从 `drivers.json` 里读 `cost_per_km`。([GitHub][7])

# 6. Agent API 二：`query_cargo(driver_id, latitude, longitude)`

这是最容易被误解的接口。

它不是免费查询。
**每调用一次，都会根据返回货源数量推进仿真时间。**

## 6.1 输入

```python
query_cargo(
    driver_id: str,
    latitude: float,
    longitude: float
) -> dict[str, Any]
```

在 baseline 里是：

```python
cargo_resp = self._api.query_cargo(
    driver_id=driver_id,
    latitude=lat,
    longitude=lng,
)
```

## 6.2 输出

返回：

```python
{
  "driver_id": driver_id,
  "items": [
    {
      "distance_km": ...,
      "cargo": {
        "cargo_id": ...,
        "price": ...,
        "start": {"lat": ..., "lng": ...},
        "end": {"lat": ..., "lng": ...},
        "load_time": [...],
        "cost_time_minutes": ...
      }
    },
    ...
  ]
}
```

其中：

| 字段                  | 含义           |
| ------------------- | ------------ |
| `distance_km`       | 查询点到货源起点的距离  |
| `cargo`             | 货源原始信息       |
| `cargo.price`       | 已经从“分”转换成“元” |
| `load_time`         | 装货时间窗        |
| `cost_time_minutes` | 干线运输耗时       |

## 6.3 内部处理

调用链：

```text
EmbeddedDecisionEnvironment.query_cargo()
  → simulation_actions.query_cargo()
    → CargoRepository.nearest_pickup_km()
  → simulation_actions.apply_cargo_query_scan_cost()
```

具体过程：

### 第一步：记录查询前仿真时间

```python
sim_min_before = manager.get_simulation_progress_minutes()
```

### 第二步：查附近货源

```python
raw = simulation_actions.query_cargo(
    repo,
    manager,
    driver_id,
    latitude,
    longitude,
    k=nearest_cargo_limit,
)
```

默认：

```python
nearest_cargo_limit = 100
```

`simulation_actions.query_cargo()` 会：

```text
1. manager.ensure_active_driver(driver_id)
2. 读取当前仿真分钟
3. 调用 repo.nearest_pickup_km(latitude, longitude, current_time_minutes, k)
4. 对返回货源 price 做 /100，变成元
5. 返回 {"driver_id": driver_id, "items": items}
```

### 第三步：根据浏览货源数量扣时间

```python
apply_cargo_query_scan_cost(
    repo,
    manager,
    driver_id,
    len(items),
    cargo_view_batch_size=10,
)
```

内部逻辑是：

```python
query_cost_minutes = ceil(items_count / cargo_view_batch_size)
```

默认 `cargo_view_batch_size = 10`。所以：

|  返回货源数 |  查询耗时 |
| -----: | ----: |
|      0 |  0 分钟 |
|   1-10 |  1 分钟 |
|  11-20 |  2 分钟 |
| 91-100 | 10 分钟 |

如果耗时大于 0：

```text
manager.advance_progress(driver_id, query_cost_minutes)
repo.sync_time_minutes(current_progress)
```

也就是说，查询不仅消耗司机时间，还会触发货源池时间同步：新货源可能上线，旧货源可能过期。([GitHub][5])

## 6.4 关键后果

baseline 当前逻辑是：

```python
status = get_driver_status()
cargo_resp = query_cargo()
prompt = build_prompt(status, cargo_resp)
```

但是 `query_cargo()` 之后，仿真时间已经推进了。
baseline 没有重新 `get_driver_status()`，所以 prompt 里的 `simulation_progress_minutes` 是查询前的旧时间。

如果你做规划器，应该改成：

```python
status0 = get_driver_status(driver_id)
cargo_resp = query_cargo(driver_id, lat, lng)
status1 = get_driver_status(driver_id)  # query 后重新取真实时间
```

这点非常重要。

# 7. `CargoRepository` 如何管理货源？

货源不是一次性全都 online。仓库维护两级池：

```text
pending：还没到 create_time 的货源
online：当前已经上架且未过期的货源
```

## 7.1 `CargoRepository.load()`

### 输入

无显式输入，使用初始化时传入的：

```python
dataset_path: Path
```

### 输出

无返回值，更新内部状态。

### 内部处理

它会：

```text
1. 打开 cargo_dataset.jsonl
2. 逐行读取货源 JSON
3. 校验 cargo_id 非空且不重复
4. 把 create_time/remove_time 转成仿真分钟
5. 如果 remove_time < create_time，则修正为 create_time
6. 按 create_minutes 排序，存入 pending
7. 清空 online
8. sync_time_minutes(0)
```

所以仿真开始时，只会把 `create_time <= 0` 且未过期的货源放入 online。([GitHub][8])

## 7.2 `sync_time_minutes(current_time_minutes)`

### 输入

```python
current_time_minutes: int
```

### 输出

无返回值，更新内部 `pending/online`。

### 内部处理

它做两件事：

```text
1. 把 create_time <= 当前时间的 pending 货源移动到 online
2. 把 remove_time < 当前时间的 online 货源移除
```

注意它有一个保护：

```python
if current_time_minutes < self._current_time_minutes:
    return
```

也就是说，货源仓库时间只会向前同步，不会倒退。

## 7.3 `nearest_pickup_km(latitude, longitude, current_time_minutes, k)`

### 输入

| 参数                     | 含义      |
| ---------------------- | ------- |
| `latitude/lng`         | 查询点     |
| `current_time_minutes` | 当前仿真时间  |
| `k`                    | 最多返回多少条 |

### 输出

```python
list[tuple[distance_km, cargo_dict]]
```

### 内部处理

它会：

```text
1. sync_time_minutes(current_time_minutes)
2. 重建 online 坐标 cache
3. 对所有 online 货源起点计算 Haversine 距离
4. 用 numpy.argpartition 找最近 k 条
5. 按距离排序返回
```

所以 `query_cargo` 返回的是“当前 online 货源中，离查询点最近的最多 100 条”，不是全局最优 100 条。([GitHub][8])

# 8. Agent API 三：`model_chat_completion(payload)`

这个接口让 Agent 调大模型。

## 8.1 输入

```python
payload: dict[str, Any]
```

baseline 传入的是 OpenAI Chat Completions 风格：

```python
{
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": prompt},
  ],
  "response_format": {"type": "json_object"},
}
```

没有传 `model` 也可以，因为网关客户端会自动补默认 model。

## 8.2 输出

返回模型原始 JSON：

```python
{
  "choices": [
    {
      "message": {
        "content": "{\"action\":\"take_order\", ...}"
      }
    }
  ],
  "usage": {
    "prompt_tokens": ...,
    "completion_tokens": ...,
    "total_tokens": ...,
    ...
  }
}
```

## 8.3 内部处理

调用链：

```text
EmbeddedDecisionEnvironment.model_chat_completion()
  → ModelGatewayClient.chat_completion()
    → requests.Session.post(...)
```

具体步骤：

```text
1. ModelGatewayClient._build_payload(payload)
   - 复制 payload
   - 如果没有 model，补默认模型名

2. ModelGatewayClient._build_headers()
   - Content-Type: application/json
   - Authorization: Bearer <api_key>

3. POST 到 model_api_url

4. resp.raise_for_status()

5. resp.json()

6. 提取 usage:
   prompt_tokens
   completion_tokens
   total_tokens
   completion_tokens_details.reasoning_tokens

7. 累加到 _last_model_usage
```

`ModelGatewayClient` 是 OpenAI 兼容 Chat Completions 请求封装；它不理解业务，只负责发 HTTP 请求和返回响应。([GitHub][9])

## 8.4 关键点：token 统计由 server 覆盖

`EmbeddedAgentDecisionEngine.decide()` 会：

```python
self._environment.reset_last_model_usage()
action = self._decision_service.decide(driver_id)
action["model_usage"] = self._environment.get_last_model_usage()
```

也就是说，哪怕选手在返回 action 里自己写了 `model_usage`，server 也会强制覆盖。真实 token 只来自本轮 `model_chat_completion()` 的累计 usage。([GitHub][5])

# 9. Baseline Agent：`ModelDecisionService.decide(driver_id)`

这是选手当前主要修改点。

## 9.1 输入

```python
driver_id: str
```

由 server 主循环传入。

## 9.2 输出

必须返回：

```python
{
  "action": "take_order" | "reposition" | "wait",
  "params": {...}
}
```

server 后续会自动补：

```python
"model_usage": {...}
```

## 9.3 内部处理

baseline 当前逻辑是：

```text
1. get_driver_status(driver_id)
2. 取当前位置 lat/lng
3. query_cargo(driver_id, lat, lng)
4. 取 items
5. 构造 prompt
6. 调 model_chat_completion()
7. 解析模型返回 JSON action
8. 返回 action
```

代码里 `_build_prompt()` 只取前 20 条候选货源，并放入：

```python
{
  "driver_id": ...,
  "simulation_progress_minutes": ...,
  "driver_status": {
    "current_lat": ...,
    "current_lng": ...,
    "truck_length": ...,
    "completed_order_count": ...
  },
  "cargo_candidates": [
    {
      "cargo_id": ...,
      "price": ...,
      "cost_time_minutes": ...,
      "load_time": ...,
      "start": ...,
      "end": ...,
      "distance_km": ...
    }
  ]
}
```

注意：没有把 `preferences` 放进 prompt。也没有收益累计、成本、历史路线、偏好状态。([GitHub][10])

# 10. `_parse_action(model_resp)`

这是 baseline 把模型输出转成系统动作的地方。

## 10.1 输入

```python
model_resp: dict[str, Any]
```

来自 `model_chat_completion()`。

## 10.2 输出

标准 action dict：

```python
{"action": "take_order", "params": {"cargo_id": "..."}}
```

或：

```python
{"action": "reposition", "params": {"latitude": ..., "longitude": ...}}
```

或：

```python
{"action": "wait", "params": {"duration_minutes": ...}}
```

## 10.3 内部处理

它会：

```text
1. 检查 choices 是非空 list
2. 读取 choices[0].message.content
3. json.loads(content)
4. 检查 action 是否属于 take_order / reposition / wait
5. 检查 params 是 dict
6. 按 action 类型进一步校验：
   - take_order: cargo_id 非空
   - reposition: latitude/longitude 可转 float
   - wait: duration_minutes 是正整数
7. 返回规范化后的 action
```

如果模型输出 markdown、解释文字、非法 JSON、缺字段，都会抛异常。baseline 没有 fallback，所以会导致本轮评测失败。([GitHub][10])

# 11. 主循环：`SimulationOrchestrator.run()`

这是评测真正跑每个司机的地方。

## 11.1 输入

来自构造函数：

| 参数                             | 作用         |
| ------------------------------ | ---------- |
| `cargo_repository`             | 货源池        |
| `driver_state_manager`         | 司机状态       |
| `agent_decision`               | Agent 决策引擎 |
| `results_dir`                  | 输出目录       |
| `reposition_speed_km_per_hour` | 空驶速度       |
| `simulation_max_steps`         | 最大 step    |
| `simulation_duration_days`     | 仿真天数       |

`run(max_steps=None)` 还可以临时覆盖最大步数。

## 11.2 输出

返回 summary：

```python
{
  "completed_steps": total_steps,
  "remaining_cargo_count": self._repo.size,
  "simulation_progress_minutes": ...,
  "simulation_wall_time": ...,
  "simulate_time_seconds": ...,
  "driver_completed_steps": ...,
  "driver_result_files": ...
}
```

并写文件：

```text
demo/results/actions_202603_<driver_id>_<timestamp>.jsonl
demo/results/run_summary_202603.json
```

## 11.3 内部处理：司机循环

核心结构是：

```python
driver_ids = manager.list_driver_ids()

for driver_id in driver_ids:
    manager.load()
    repo.load()
    manager.start_simulation(driver_id=driver_id, progress_minutes=0)
    repo.sync_time_minutes(0)

    while total_steps < max_total_steps
          and driver_progress_minutes < month_duration_minutes
          and repo.size > 0:
        ...
```

这里有一个非常关键的设计：

> 每个司机开始前，`manager.load()` 和 `repo.load()` 会重新加载。
> 所以每个司机面对的是独立的一个月仿真，不是多个司机抢同一批货源。

但是 `ModelDecisionService` 对象是在外面构造的，不会每个司机重建。
所以如果你在 Agent 里加状态缓存，必须按 `driver_id` 隔离，或者在检测到某司机时间回到 0 时重置该司机状态。([GitHub][3])

# 12. 每一个 step 如何运行？

每轮循环大致是：

```python
step_start_minutes = manager.get_simulation_progress_minutes()
before_status = manager.get_driver_status(driver_id)

action = self._call_agent(driver_id)

progress_after_decision = manager.get_simulation_progress_minutes()
query_scan_cost_minutes = progress_after_decision - step_start_minutes

result = self._apply_action(driver_id, action)

after_status = manager.get_driver_status(driver_id)
current_progress = manager.get_simulation_progress_minutes()

step_elapsed_minutes = current_progress - step_start_minutes
action_exec_cost_minutes = current_progress - progress_after_decision
```

这段逻辑很重要。它把每个 step 拆成两段：

```text
step_elapsed_minutes
= query_scan_cost_minutes + action_exec_cost_minutes
```

其中：

| 阶段     | 谁造成                        | 是否推进时间           |
| ------ | -------------------------- | ---------------- |
| 决策阶段   | Agent 调用 `query_cargo`     | 会推进              |
| 动作执行阶段 | wait/reposition/take_order | 会推进              |
| 模型调用   | `model_chat_completion`    | 不推进仿真时间，只计 token |

最终每轮写入 action log：

```python
{
  "step": ...,
  "driver_id": ...,
  "step_elapsed_minutes": ...,
  "query_scan_cost_minutes": ...,
  "action_exec_cost_minutes": ...,
  "position_before": ...,
  "position_after": ...,
  "simulation_end_time": ...,
  "action": action,
  "token_usage": token_usage,
  "result": result,
}
```

后面的收益脚本会基于这个 JSONL 重新校验时间和位置。([GitHub][11])

# 14. `_apply_action(driver_id, action)`

这是系统执行 Agent 动作的地方。

## 14.1 输入

```python
driver_id: str
action: dict[str, Any]
```

action 必须形如：

```python
{
  "action": "...",

}
```

## 14.2 输出

执行结果 dict。不同动作返回不同字段。

## 14.3 内部处理

它先做：

```python
action_name = str(action.get("action", "")).strip().lower()
params = action.get("params", {})
```

如果 `params` 不是 dict，直接报错。

然后三类分发：

## 14.4 `wait`

输入：

```python
{
  "action": "wait",
  "params": {"duration_minutes": N}
}
```

内部：

```python
duration_minutes = int(params.get("duration_minutes", 1))
simulation_actions.wait(repo, manager, driver_id, duration_minutes)
```

如果没传 `duration_minutes`，默认 1 分钟。

## 14.5 `reposition`

输入：

```python
{
  "action": "reposition",
  "params": {
    "latitude": ...,
    "longitude": ...
  }
}
```

内部：

```python
target_lat = float(params["latitude"])
target_lng = float(params["longitude"])
simulation_actions.reposition(...)
```

如果缺 `latitude/longitude`，会异常。

---

## 14.6 `take_order`

输入：

```python
{
  "action": "take_order",
  "params": {"cargo_id": "..."}
}
```

内部：

```text
1. cargo_id = params["cargo_id"]
2. repo.get_by_id(cargo_id)
3. 如果 cargo 不存在：
   - manager.advance_progress(driver_id, 1)
   - repo.sync_time_minutes(progress)
   - 返回 accepted=False
4. 如果 cargo 存在：
   - 调 simulation_actions.take_order(...)
5. 如果 take_order 抛 ValueError：
   - 同样推进 1 分钟
   - 返回 accepted=False
```

所以接一个已经失效的 cargo，不会导致程序中断，而是消耗 1 分钟并返回失败。([GitHub][11])

# 16. 动作二：`simulation_actions.reposition()`

## 16.1 输入

```python
reposition(
    repo,
    manager,
    driver_id,
    latitude,
    longitude,
    speed_km_per_hour,
)
```

## 16.2 输出

```python
{
  "current_lat": latitude,
  "current_lng": longitude,
  "simulation_progress_minutes": ...,
  "simulation_wall_time": ...,
  "distance_km": ...
}
```

## 16.3 内部处理

```text
1. current = manager.get_driver_status(driver_id)
2. 用 Haversine 计算当前位置到目标点距离
3. distance_to_minutes(distance, speed) 得到耗时
4. manager.update_driver_position(driver_id, target)
5. manager.advance_progress(driver_id, duration_minutes)

7. 返回新位置、时间、空驶距离
```

注意 `distance_to_minutes()` 里如果距离小于等于 0，也返回 1 分钟；否则按 `ceil(distance / speed * 60)` 向上取整。([GitHub][12])

# 17. 动作三：`simulation_actions.take_order()`

这是最复杂的动作。

## 17.1 输入

```python
take_order(
    repo,
    manager,
    driver_id,
    cargo_id,
    reposition_speed_km_per_hour,
    simulation_horizon_minutes=None,
)
```

## 17.2 输出

成功时大致返回：

```python
{
  "accepted": True,
  "detail": "接单后已完成、推进时间并更新到卸货地",
  "driver_id": ...,
  "cargo_id": ...,
  "simulation_progress_minutes": ...,
  "simulation_wall_time": ...,
  "pickup_deadhead_km": ...,
  "haul_distance_km": ...,
  "income_eligible": True | False
}
```

如果到达装货点时已经错过装货窗：

```python
{
  "accepted": False,
  "detail": "load_time_window_expired",
  "driver_id": ...,
  "cargo_id": ...,
  "simulation_progress_minutes": ...,
  "simulation_wall_time": ...,
  "pickup_deadhead_km": ...,
  "haul_distance_km": 0
}
```

## 17.3 内部处理顺序

完整流程是：

```text
1. repo.get_by_id(cargo_id)
   - 不存在则抛 ValueError

2. 读取货源 start 坐标

3. 读取司机当前位置

4. 计算当前位置到装货点的空驶距离 distance_pickup_km

5. 如果传入 simulation_horizon_minutes：
   - 预估成功完单时间
   - 如果会超过仿真上界，则 income_eligible=False
   - 注意：仍允许执行，只是收入结算可能不计

6. 如果 distance_pickup_km > 0：
   - 计算 pickup_minutes
   - 更新司机位置到装货点
   - 推进 pickup_minutes
   - repo.sync_time_minutes()

7. 解析 load_time 时间窗

8. 如果到达时间 > load_time_end：
   - repo.remove_by_id(cargo_id)
   - 返回 accepted=False
   - 注意：司机已经空驶到了装货点，时间也已经推进

9. 如果到达时间 < load_time_start：
   - 等到 load_time_start
   - 推进 wait_minutes
   - repo.sync_time_minutes()

10. 读取 cost_time_minutes

11. 读取 end 坐标

12. 调用 manager.take_order()
    - 增加 completed_order_count
    - 推进 duration_minutes
    - 更新司机位置到卸货点

13. repo.remove_by_id(cargo_id)

14. 计算 start 到 end 的 haul_distance_km

15. 返回结果
```

这说明 `take_order` 不是一个瞬时动作，而是：

```text
空驶到装货点
→ 必要时等待装货窗
→ 干线运输
→ 到达卸货点
```

这也说明如果你本地规划时没有复刻这套逻辑，就会严重误判收益和时间。([GitHub][12])

# 18. `DriverStateManager` 如何推进状态？

`DriverStateManager` 是司机状态的唯一来源。

## 18.1 `load()`

输入：无显式输入，使用 `drivers_path`。
输出：无返回值。

内部：

```text
1. 读取 drivers.json
2. 校验是非空数组
3. 检查每项有 driver_id
4. 建立 driver_id → profile
5. 初始化：
   - current_driver_id
   - simulation_started=False
   - simulation_progress_minutes=0

   - taken_cargo_ids
   - completed_orders_by_driver
```

## 18.2 `start_simulation(driver_id, progress_minutes)`

输入：

```python
driver_id: str | None
progress_minutes: int
```

输出：

```python
get_system_state()
```

内部：

```text
1. 校验 progress_minutes >= 0
2. 选中 current_driver_id
3. simulation_started=True
4. simulation_progress_minutes=progress_minutes
5. 返回系统状态
```

## 18.3 `ensure_active_driver(driver_id)`

内部检查：

```text
1. driver_id 是否存在
2. simulation 是否已开始
3. driver_id 是否等于 current_driver_id
```

这意味着在模拟 D001 时，你不能操作 D002。

输入：

```python
duration_minutes
```

输出：

```python
新的 simulation_progress_minutes
```

内部：

```text
1. ensure_active_driver
2. 校验 duration_minutes >= 0
3. simulation_progress_minutes += duration_minutes
4. 返回新时间
```

## 18.5 `update_driver_position(driver_id, latitude, longitude)`

输入：目标经纬度。
输出：无。

内部：

```text
1. ensure_active_driver
2. 修改 profile["current_lat/current_lng"]
```

## 18.6 `take_order(...)`

这是低层状态更新，不负责装货窗和空驶，外层 `simulation_actions.take_order()` 已经处理了那些。

输入：

```python
driver_id,
cargo_id,
duration_minutes,
end_latitude,
end_longitude
```

输出：

```python
{
  "accepted": True,
  "detail": "...",
  "driver_id": ...,
  "cargo_id": ...,
  "simulation_progress_minutes": ...,
  "simulation_wall_time": ...
}
```

内部：

```text
1. ensure_active_driver
2. 检查 cargo_id 非空
3. 检查 duration_minutes >= 0
4. 检查 cargo_id 没被 taken
5. 加入 taken_cargo_ids
6. completed_order_count += 1

```

所以真正完成订单数量是在这里增加的。([GitHub][6])

# 19. 每步日志为什么重要？

仿真结束后，系统不是直接相信运行时的收入。
它会写 action JSONL，然后 `calc_monthly_income.py` 重新计算收益和合法性。

每条 action log 包括：

```python
{
  "step_elapsed_minutes": ...,
  "query_scan_cost_minutes": ...,
  "action_exec_cost_minutes": ...,
  "position_before": ...,
  "position_after": ...,
  "action": ...,
  "token_usage": ...,
  "result": ...
}
```

收益脚本会验证：

```text
1. step_elapsed == query_scan_cost + action_exec_cost
2. end_minutes - prev_end_minutes == step_elapsed
3. wait 不改变位置
4. reposition 后位置等于目标点
5. reposition 耗时等于 Haversine 距离 / speed
6. take_order 的 cargo_id 存在
7. 接单时点在货源 create/remove 有效期内
8. 成功接单后位置必须等于货源 end
9. 接单耗时必须等于：
   pickup_minutes + load_window_wait + cost_time_minutes
10. 收入只统计 horizon 内完成的订单
```

如果某个司机的 action log 校验失败，该司机收入会被置零并记录 validation_error。([GitHub][7])

# 20. 收益计算怎么做？

收益脚本先读：

```text
cargo_dataset.jsonl
drivers.json
config.json
run_summary_202603.json
actions_202603_*.jsonl
```

然后对每个司机计算：

```python
gross_income = 完成订单价格之和
distance_km = 空驶距离 + 干线距离 + reposition 距离
cost = distance_km * cost_per_km
net_income = gross_income - cost - preference_penalty
```

其中 `cost_per_km` 来自 `drivers.json`。货源价格原始单位是分，收益脚本会 `/100` 转成元。([GitHub][7])

# 21. 偏好罚分怎么计算？

收益脚本里偏好规则当前是针对 D006-D010 写死的：

| 司机   | 偏好规则                           |
| ---- | ------------------------------ |
| D006 | 每天至少连续休息 5 小时                  |
| D007 | 23:00-04:00 不接单、不空驶            |
| D008 | 每月至少 2 天完全不出车                  |
| D009 | 每天 23:00 前回家，次日 08:00 前不再接单/空驶 |
| D010 | 每月至少 5 天到目标点；禁止进入指定区域          |

它会从 action log 里按动作区间判断是否违规。比如 D007 是检查 `take_order/reposition` 的动作执行区间是否和 23:00-04:00 重叠；D008 是检查每天是否有 `take_order/reposition`；D006 是用 `wait` 区间近似休息时段。([GitHub][7])

这也说明：
如果你想满足偏好，不能只在 prompt 里说“注意休息”，而必须让返回的动作日志真的体现出 wait/reposition/take_order 的时间区间满足规则。

一次 step 的完整链路如下：

```text

  before_status = manager.get_driver_status(driver_id)

  action = _call_agent(driver_id)

    EmbeddedAgentDecisionEngine.decide(driver_id)

      environment.reset_last_model_usage()

      ModelDecisionService.decide(driver_id)

          DriverStateManager.get_driver_status()

        cargo_resp = api.query_cargo(driver_id, lat, lng)

          simulation_actions.query_cargo()
            CargoRepository.nearest_pickup_km()
              CargoRepository.sync_time_minutes()
              计算 online 货源距离
              返回最近 K 条

          simulation_actions.apply_cargo_query_scan_cost()
            manager.advance_progress()
            repo.sync_time_minutes()

        model_resp = api.model_chat_completion(payload)

          ModelGatewayClient.chat_completion()
            POST model_api_url

          environment 累加 usage

        action = _parse_action(model_resp)

      server 覆盖 action["model_usage"]

      return action

  progress_after_decision = 当前仿真分钟
```

```text
返回 action 后，server 才真正执行 wait/reposition/take_order。
```

# 23. 对你后续改算法最重要的接口约束

你设计规划器时，必须遵守这些事实：

## 23.1 不要改 server 入口

保留：

```python
class ModelDecisionService:
    def __init__(self, api: SimulationApiPort): ...
    def decide(self, driver_id: str) -> dict[str, Any]: ...
```

因为 server 是这样 import 的：

```python
from agent.model_decision_service import ModelDecisionService
```

## 23.2 `query_cargo` 之后必须重新取状态

因为查询会推进时间。正确写法：

```python
status0 = self._api.get_driver_status(driver_id)
cargo_resp = self._api.query_cargo(driver_id, lat, lng)
status1 = self._api.get_driver_status(driver_id)
```

## 23.3 不能在一个 step 内随便多次 query

每次 query 都会消耗：

```python
ceil(items_count / 10)
```

分钟。查询 100 条就是 10 分钟。多查几次，司机一天可能就被你查没了。

## 23.4 Agent 内部状态要按司机隔离

虽然每个司机仿真前 `manager/repo` 会 reload，但 `ModelDecisionService` 对象可能跨司机持续存在。
所以你的缓存应该是：

```python
memory_by_driver[driver_id]
```

不能用一个全局 memory 混在一起。

## 23.5 动作返回只认三类

```python
{"action": "take_order", "params": {"cargo_id": "..."}}
{"action": "reposition", "params": {"latitude": ..., "longitude": ...}}
{"action": "wait", "params": {"duration_minutes": ...}}
```

多余解释没用；非法字段可能导致异常。

# 24. 一句话总结

这个仿真系统的本质是：

> server 进程内加载你的 `ModelDecisionService`，每个 step 调一次 `decide(driver_id)`；你在 `decide()` 里通过注入的 `SimulationApiPort` 获取司机状态、查询货源、调用模型；其中 `query_cargo` 会真实推进仿真时间，模型调用只计 token；你返回一个动作 JSON 后，`SimulationOrchestrator` 执行动作、推进司机位置和时间、写入 action log；最后收益脚本不信任你的自报收入，而是根据 action log 重新校验时间、位置、订单有效期、偏好罚分并计算最终净收益。

所以后续做规划器时，最关键不是“让大模型输出动作”，而是**在 `ModelDecisionService.decide()` 这个固定接口内，正确使用 `get_driver_status/query_cargo/model_chat_completion`，并返回能被 `_apply_action()` 和收益脚本校验通过的动作**。
